## What it is

The Idk TRAINING data from Cheng et al. (ICML 2024) used to align
Llama-2-7b-chat with Idk-SFT and Idk-DPO. This is distinct from the method
OUTPUTS in `datasets/say-i-dont-know-outputs/` (those are test generations; this
is the training input). Our Phase 1 bridge arm trains on this exact data so its
over-refusal numbers are directly comparable to Cheng's published 42.71%
(Idk-SFT) / 23.27% (Idk-DPO).

## How to obtain (one authorized, user-run command)

Network egress to Google Drive + GitHub is required and is authorized for THIS
dataset only:

```
SSL_CERT_FILE=$(python3 -m certifi) \
  python datasets/scripts/fetch_datasets.py --only say-i-dont-know-training
```

This (1) gdowns the pinned Drive file id `1xN-xtx12eHL-1-pIsS5-vERXrgzfMnw9` to
`data.zip` and records its sha256, (2) clones `OpenMOSS/Say-I-Dont-Know` at the
pinned commit `a4768458638fa78a9a43252d319514db425dcaf3`, (3) unzips and asserts
the layout `sft_data/llama-2-7b-chat/triviaqa_{train,dev}_tp1.0_10responses_with_em_labels.json`
(a mismatch is a hard stop, not a silent guess), (4) runs their
`process_sft_data.py --model_name llama-2-7b-chat --threshold 1.0` to regenerate
the Idk-SFT train/valid/test sets, and locates the Idk-DPO preference-pairs file
shipped in the zip. All outputs land under this gitignored directory.

## Splits and per-file provenance (two distinct origins)

The two arms have DIFFERENT provenance and this distinction is recorded
honestly: Idk-SFT is REGENERATED by us from the zip's probe outputs; Idk-DPO
pairs are VENDORED AS-SHIPPED from the zip unchanged. Each file carries its own
sha256 (filled after the authorized fetch) so the origin and integrity of each
is independently checkable.

### Idk-SFT (REGENERATED via process_sft_data.py @ pinned commit)

Produced by running OpenMOSS `Idk_datasets/process_sft_data.py` (cloned at
pinned commit `a4768458638fa78a9a43252d319514db425dcaf3`)
`--model_name llama-2-7b-chat --threshold 1.0` over the zip's per-question
10-response EM-labeled probe outputs. 90/10 train/valid split as in their
script. These files are OUR derived output, not OpenMOSS's shipped bytes.

### Idk-DPO (VENDORED as-shipped from data.zip)

The preference pairs ship inside `data.zip` unchanged; the OpenMOSS repo has NO
from-scratch pair builder (`process_preference_data.py` only reformats existing
pairs for PPO reward modeling, so it is NOT used here). These are OpenMOSS's
shipped bytes, vendored as-is.

| File | Origin | Items / pairs | sha256 |
|---|---|---|---|
| `triviaqa_train_and_valid_llama2_7b_chat_threshold_1.0_preference_pairs.json` | vendored as-shipped | train 29,847 items / 131,945 pairs; test 3,316 items / 14,527 pairs | `99d86ec1ff89b2eea3d7d4f81a9a58430ea76d7e30834ba47452ea9f2e06cd20` |

The DPO file is a single JSON object with `train` and `test` keys; each entry is
one question carrying multiple `preference_pairs`, so the pair count exceeds the
question-item count.

### Verification vs Cheng et al. 2401.13275

- **EXACT MATCH:** the Idk-SFT test split is 11,313 samples, exactly the paper's
  n=11,313 TriviaQA Idk test set (the same n our `cheng_test_gold.jsonl` and the
  meta-analysis reanalysis use). This confirms the regeneration reproduces the
  paper's test partition.
- The train pool before the script's 90/10 split is 78,859 + 8,763 = 87,622
  questions, consistent with the paper's full-TriviaQA training source (the paper
  reports method-level over-refusal rates rather than a single train-set n, so the
  train/valid counts are verified structurally, not against a single printed
  number).
- The Idk-SFT known/unknown balance (train: 35,996 answered vs 42,863 refusal
  targets) reflects Llama-2-7b-chat's measured answerability at Ik threshold 1.0,
  as expected for the abstention-training setup.
